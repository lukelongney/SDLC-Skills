S1 — Concept Brief

solid #ccc Interactive Skill

Position: First skill — initiates the chain

Purpose

Collaborative problem definition and scope identification. Helps stakeholders articulate the business problem, identify affected areas, and establish project boundaries.

 Inputs
  • User's initial idea or problem statement
  • Business context from conversation
  • Optional: Confluence/Jira discovery via MCP
 Outputs
  • Primary: docs/concept-briefs/YYYY-MM-DD-<topic>-concept-brief.md
  • YAML frontmatter as context summary for downstream skills
 Key Sections
SectionDescription
Problem StatementWhat problem are we solving?
Affected Business AreasWho is impacted?
Proposed ScopeWhat's in and out?
Success CriteriaHow do we know we succeeded?
Initial RisksWhat could go wrong?

S2 — Feasibility Assessment

solid #ccc Interactive Skill

Position: Second skill — receives S1

Purpose

Structured viability analysis covering technical, operational, economic, and scheduling dimensions. Produces go/no-go recommendation with ROM estimates.

 Assessment Dimensions
DimensionFocus
TechnicalCan we build it?
OperationalCan we run it?
EconomicIs it worth it?
ScheduleCan we deliver in time?
Outputs
  • Primary: docs/feasibility/YYYY-MM-DD-<topic>-feasibility.md
  • ROM estimates and risk assessment
  • Go/Conditional Go/No-Go recommendation

S3 — High Level Requirements

solid #ccc Interactive Skill

Position: Third skill — iterates through v1, v2, v3

Purpose

Structured requirements elicitation producing capability-grouped requirements with MoSCoW prioritisation. Iterates based on architecture (v2) and security (v3) feedback.

 Version Evolution
VersionTriggerAdditions
v1After S2 approvalInitial requirements
v2After S4 architectureTechnical constraints, refined NFRs
v3After S5 threat modelSecurity requirements
Outputs
  • Primary: docs/requirements/YYYY-MM-DD-<topic>-requirements.md
  • Capability groups with functional and non-functional requirements
  • MoSCoW prioritisation and complexity ratings

  • Feedback for S3 v2

S5 — Threat Modelling

solid #ccc Interactive Skill

Position: Fifth skill — security-focused

Purpose

Collaborative threat modelling using STRIDE methodology and OWASP ASVS control mapping. Produces security requirements for S3 v3.

 STRIDE Categories
CategoryThreat Type
SSpoofing — Can attacker impersonate?
TTampering — Can attacker modify data?
RRepudiation — Can attacker deny actions?
IInformation Disclosure — Can attacker access data?
DDenial of Service — Can attacker disrupt?
EElevation of Privilege — Can attacker gain access?
Outputs
  • Primary: docs/threat-models/YYYY-MM-DD-<topic>-threat-model.md
  • STRIDE analysis per component
  • OWASP ASVS control mapping
  • Security requirements addendum for S3 v3

S6 — Design Validation

solid #ff9900 Agentic Skill

Position: Sixth skill — autonomous gate before Jira

Purpose

Autonomous cross-validation of all design artifacts (S1-S5). Checks traceability, completeness, consistency, and pattern compliance. Must pass before Jira scaffolding.

 Validation Categories
CategoryChecks
TraceabilityScope → Requirements → Architecture → Threats
CompletenessAll required sections present
ConsistencyNo contradictions between artifacts
Pattern ComplianceTechnologies against approved patterns
Verdicts
VerdictMeaningNext Step
GreenReadyAll checks passProceed to S7
YellowReady with ConditionsWarnings onlyProceed with noted conditions
RedNot ReadyCritical issuesResolve before continuing
Output
  • Primary: docs/validation/YYYY-MM-DD-<topic>-design-validation.md

S7 — Jira Project Scaffolding

solid #ccc Interactive Skill

Position: Seventh skill — translates design to work items

Purpose

Translates validated design artifacts into Jira epics and stories. Creates technical enablers, security stories, and user stories with acceptance criteria.

 Story Types
TypeSourceExample
User StoriesS3 Requirements"As a user, I can..."
Technical EnablersS4 ArchitectureInfrastructure, frameworks
Security StoriesS5 Threat ModelControl implementations
Outputs
  • Jira epics aligned to capability groups
  • Stories with acceptance criteria
  • Technical enabler stories from ADRs
  • Security stories from threat mitigations
  • Local scaffold: docs/jira/YYYY-MM-DD-<topic>-scaffolding.md

S8 — Security Assessment

solid #ff9900 Agentic Skill

Position: Eighth skill — independent verification of design artifacts

Purpose

Independent security assessment of design artifacts (S4, S5, S7) against industry frameworks. Framework tiering by project classification with user override. Spot checks high-priority threats have Jira stories. Produces verdict and optional Jira tickets for gaps.

 Framework Tiering
ClassificationDefault Frameworks
StandardOWASP Top 10 2025, ASVS Level 1
ElevatedAbove + ASVS Level 2 + SOC 2
CriticalAbove + ASVS Level 3 + ISO 27001 + MITRE ATT&CK

User can add or remove frameworks before assessment runs.

 What S8 Checks
FrameworkAssessment Focus
OWASP Top 10Each category addressed in S4 security architecture
OWASP ASVSS5 controls cover required sections for level
SOC 2Trust Service Criteria in design
ISO 27001Relevant Annex A controls designed
MITRE ATT&CKDetection/logging for relevant techniques
Verdicts
VerdictCriteriaAction
GreenApprovedNo Critical or High findingsProceed to S9
YellowApproved with ConditionsHigh findings documentedProceed with conditions
RedNot ApprovedCritical findings presentResolve before continuing
Jira Integration
  • Spot checks 3-5 high-priority threats from S5 have Jira stories
  • Offers to create defect tickets for gaps (user confirms)
  • Uses same Jira integration as S7
 Output
  • Primary: docs/security/YYYY-MM-DD-<topic>-security-assessment.md
  • Optional: Jira defect tickets for gaps

S9 — Operational Readiness & Handoff

solid #ff9900 Agentic Skill

Position: Ninth and final skill — consolidates design for delivery

Purpose

Consolidates the complete design phase into a handoff package for delivery. Assesses change impact (technical + organizational) and operational readiness. Produces audience-specific sections. Informs delivery — not a blocking gate.
